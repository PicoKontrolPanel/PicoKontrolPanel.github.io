Raspberry Pi Pico W and send a command (LED on/off) on both Android (Chrome) and

GOAL
-------------------------------------------------------------------------------
Recreate, in the React app, the same flow / function / usability as the Unity app
described in DESIGN_HANDOFF.md:
  - First-run username creation (stored locally).
  - Dashboard: "Mine Enheder" / "Andre Enheder" swipeable device cards + a way to
    find/connect new devices.
  - Connection-progress screen driven by the BLE handshake.
  - Create-Device setup (icon picker, Privat/Offentlig, edit permission) for
    unowned devices.
  - Control Panel: a dot-grid of buttons/sliders that drive the robot
    (press -> "button,<NAME>", slider -> "slider,<NAME>:<VALUE>").
  - Edit Mode: drag + grid-snap, collision blocking, resize handles, rotate,
    add/delete controls, and save the layout back to the device.
  - Side menu, debugger/log modal, the various modals.

Match the VISUAL LANGUAGE: brand red #C6504E, white, big rounded touch targets,
Danish copy exactly as listed in DESIGN_HANDOFF.md. Portrait, mobile-first.

-------------------------------------------------------------------------------
HARD RULES (unchanged from the test app)
-------------------------------------------------------------------------------
  - LOCAL-ONLY: no backend, no database, no analytics, no telemetry, no network
    calls at runtime. Identity + saved devices live in browser storage only
    (localStorage/IndexedDB). Nothing about a visiting device is logged or sent.
  - CROSS-PLATFORM: must work on Android Chrome AND iOS via Bluefy (Web Bluetooth).
    Keep the existing feature-detection + "open in Bluefy on iPhone/iPad" message.
  - HTTPS / GitHub Pages hosting is fine (static site). requestDevice() must be
    called from a user gesture.
  - Reuse the existing BLE transport (GATT connect + chunked write/notify) from the
    test app. On top of it, BUILD the full protocol in reference/PROTOCOL.md — the
    handshake, the reliable-stream framing (prep/ack:prep/numbered frames/miss), and
    layout/update streaming. The test app only did LED on/off, so none of this exists
    yet; implement it using PROTOCOL.md + AppManager.cs as reference. Do NOT rebuild
    the low-level GATT transport from scratch.

-------------------------------------------------------------------------------
IMPORTANT ADAPTATIONS (Unity -> Web) — call these out, don't copy blindly
-------------------------------------------------------------------------------
  - SCANNING: Unity background-scans to fill a "nearby" list and to toggle
    green/grey "in range" dots. Web Bluetooth CANNOT do this. Replace the nearby
    list with a "Find enhed" button that calls requestDevice() (browser chooser);
    use getDevices() for quick reconnect to saved devices where supported. The
    availability dot may not be reliably knowable on web — drop or approximate it.
    Keep the card visuals; change only what populates them. (See DESIGN_HANDOFF
    section 2.3.)
  - LAYOUT/MATH: reproduce the grid model exactly (11 cols x 31 rows, edgeMargin
    50; half-step center coords centerX2/centerY2; whole-cell spans; rot 0/90/180/
    270) because it is the on-wire layout format. But implement it with DOM/CSS or
    canvas, not Unity RectTransforms.
  - ICONS: every icon/image used in the Unity scene is provided under
    reference/icons/ (device-icons/ + ui-glyphs/) and reference/art/, documented in
    reference/icons/ICONS.md. The device-icon ORDER is a contract: iconID is the
    index into that 12-item list and is sent in "create" / echoed in "owned". Reuse
    or re-export the art, but keep the indices stable.
  - Do not try to mirror Unity's GameObject tree, coroutines, or component layout.
    Translate intent into idiomatic React (components, state, hooks/refs, CSS
    transitions / a small animation lib).

-------------------------------------------------------------------------------
SUGGESTED TECH (use judgement; keep it lean and maintainable)
-------------------------------------------------------------------------------
  - Continue with Vite + React + TypeScript (same as the test app).
  - Keep BLE logic isolated in one module; add a small typed "protocol" layer that
    implements reference/PROTOCOL.md (handshake, reliable multi-line stream for
    layout/update, plain sends for button/slider).
  - State: a light store (Context/Zustand) for user, saved devices, connection
    state, current layout, edit-mode state.
  - Drag/resize: pointer events; a minimal custom implementation is fine and keeps
    the grid math under your control (a library is optional).
  - Persistence: a tiny storage module mirroring DeviceSaveSystem/UserSaveSystem.
