ROLE / CONTEXT
-------------------------------------------------------------------------------
You are helping build a SMALL proof-of-concept. There is an existing Unity
(C#) mobile app that controls Raspberry Pi Pico W robots over Bluetooth Low
Energy (BLE). Distributing that app through the Apple App Store and Google Play
is the problem we are trying to escape (store approval, Apple developer account,
parental download controls). The target users are kids on phones/tablets.

We want to find out whether a plain WEBSITE (React) can replace the native app
by talking to the Pico directly over BLE from the browser. THIS TASK IS ONLY THE
FEASIBILITY TEST — not the full rebuild.

DO read these files in this repo before coding:
  - reference/PROTOCOL.md            <- the distilled spec you must follow
  - reference/micropython-original/  <- the real Pico firmware (BLEPeripheral.py, main.py)
  - reference/unity-bridge/          <- how the native app speaks BLE (UUID/format proof)

THE GOAL (keep it minimal)
-------------------------------------------------------------------------------
Build the smallest thing that proves this pipeline works END TO END:

  A React website  --(Web Bluetooth)-->  a Raspberry Pi Pico W

Success = a phone/tablet opens the website, taps "Connect", picks the Pico, and
then tapping a few command buttons causes a visible reaction on the Pico
(onboard LED toggling and/or a serial print, and/or an optional text reply shown
in the web UI).

Explicitly IN scope:
  1. A React web app with:
       - a "Connect" button (Web Bluetooth device chooser; must be triggered by a tap)
       - a connection status indicator (disconnected / connecting / connected / error)
       - 3-5 command buttons that send commands to the Pico (e.g. STOP, FREM,
         TILBAGE, plus one slider is a nice-to-have, not required)
       - a small on-screen log showing what was sent and any bytes received back
       - a "Disconnect" button
  2. A SIMPLIFIED main.py and a SIMPLIFIED BLEPeripheral.py for the Pico W that
     advertise the correct service, accept writes, parse "button,<NAME>" lines,
     and react visibly (toggle the onboard LED, print over serial, and OPTIONALLY
     notify a short "ok:<NAME>" string back). You ARE ALLOWED to write simplified
     versions of main.py and BLEPeripheral.py — do not port the whole protocol.

Explicitly OUT of scope (do NOT build these yet):
  - The drag-and-drop control-panel editor, layouts, icons, user accounts.
  - Ownership / permissions / the HELLO / who_are_you / request_permission handshake.
  - The reliable-stream framing (prep / ack:prep / numbered "N-payload" / miss).
  - Saving devices, layout streaming, multi-robot support.

-------------------------------------------------------------------------------
HARD REQUIREMENTS
-------------------------------------------------------------------------------
PRIVACY / LOCAL-ONLY (important):
  - The website must be 100% client-side. NO backend, NO database, NO analytics,
    NO telemetry, NO cookies, NO localStorage/sessionStorage, NO network calls of
    any kind at runtime. When a device visits, nothing about it is logged, sent,
    or stored anywhere. All BLE communication happens locally, device-to-Pico.
  - The only network involved is the static hosting that serves the HTML/JS/CSS.

CROSS-PLATFORM (Android AND iOS) — READ CAREFULLY:
  - Web Bluetooth (navigator.bluetooth) is the browser API used.
  - ANDROID: works in Chrome, Edge, and Samsung Internet out of the box.
  - iOS: Apple's Safari/WebKit does NOT implement Web Bluetooth, and Apple forces
    every iOS browser (Chrome, Firefox, Edge) to use WebKit — so NONE of the
    built-in iOS browsers can do BLE, and neither can a normal PWA / "Add to Home
    Screen". This is an Apple platform limitation, not something code can fix.
  - The supported way to run this on iPhone/iPad is a third-party Web-Bluetooth
    browser that ships its own CoreBluetooth bridge: "Bluefy" or "WebBLE" (both
    free on the App Store). The user installs that browser ONCE and opens our URL
    in it. We publish nothing to Apple.
  - Therefore: the app must feature-detect Web Bluetooth and, if it is missing,
    show a clear, kid-friendly message explaining: "On iPhone/iPad, open this page
    in the free Bluefy browser" (and "On Android, use Chrome"). Do not let the app
    silently appear broken on iOS Safari.
  - The same single codebase must work on Android Chrome AND inside Bluefy on iOS.
    Do not add platform-specific BLE code paths; standard Web Bluetooth covers both.

SECURE CONTEXT:
  - Web Bluetooth only works on HTTPS (or http://localhost). To test on a real
    phone you cannot use a LAN IP over plain http. Provide a simple path to an
    HTTPS URL — recommend deploying the static build to a free host (GitHub Pages,
    Netlify, Vercel, or Cloudflare Pages), or using an https dev tunnel
    (cloudflared / ngrok). Optionally add vite-plugin-basic-ssl or mkcert for
    local https, but note self-signed certs are awkward to trust on phones.
  - requestDevice() MUST be called from a user gesture (a click/tap handler).

THE BLE CONTRACT (full details in reference/PROTOCOL.md)
-------------------------------------------------------------------------------
  Service UUID : b8e06067-62ad-41ba-9231-206ae80ab551
  Write  char  : f897177b-aee8-4767-8ecc-cc694fd5fce0   (phone -> Pico, WRITE)
  Notify char  : f897177b-aee8-4767-8ecc-cc694fd5fce1   (Pico -> phone, NOTIFY)
  Advertised name prefix: "PicoW-"  (e.g. PicoW-Test)

  Command format (UTF-8, newline-terminated):
    button : "button,<NAME>\n"          e.g. "button,STOP\n"
    slider : "slider,<NAME>:<VALUE>\n"  e.g. "slider,Speed:42\n"
  Control commands are fire-and-forget (no ACK required).

Keep these UUIDs identical on the React side and your simplified Pico firmware so
they connect. (Keeping the production UUIDs also means your test Pico stays
compatible with the existing native app, which is a bonus, not a requirement.)

SUGGESTED TECH STACK (use your judgement, keep it lean)
-------------------------------------------------------------------------------
  - Vite + React + TypeScript.
  - Add the type package "@types/web-bluetooth" for typings.
  - No UI framework needed; plain components + a little CSS is fine. Make the
    buttons big and touch-friendly (this is for kids on tablets).
  - Keep all BLE logic in one small module (e.g. src/ble.ts) with functions:
    connect(), disconnect(), sendButton(name), sendSlider(name, value),
    onMessage(cb), onStatusChange(cb). Keep React components thin.
